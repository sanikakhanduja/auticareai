🎯 HOW IT WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DOCTOR ASSIGNMENT:
  1. Frontend calls get_care_doctors_with_capacity()
  2. Displays Doc1, Doc2, Doc3 with live patient counts
  3. User selects doctor
  4. Frontend calls assign_care_doctor_to_child()
  5. Backend validates:
     ✓ Parent owns child
     ✓ Doctor exists and is valid
     ✓ Doctor has capacity (< 5 patients)
  6. If valid:
     - Remove old assignment (if exists)
     - Create new assignment
     - Update children.assigned_doctor_id
  7. Frontend refreshes and shows "Selected" badge

CAPACITY ENFORCEMENT:
  - When doctor reaches 5 patients → available = FALSE
  - Frontend disables "Select Doctor" button
  - Shows "Unavailable" badge
  - Error if parent tries to assign: "Doctor is at full capacity"

FOLLOW-UP UPLOAD RESTRICTION:
  1. Doctor sets observation_end_date when creating observation report
  2. Frontend checks: current_date >= observation_end_date
  3. Before date: Button disabled with countdown message
  4. After date: Button enabled for upload

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Check doctors exist:
  SELECT id, full_name, email 
  FROM public.profiles 
  WHERE full_name IN ('Doc1', 'Doc2', 'Doc3');

Check doctor capacity:
  SELECT * FROM get_care_doctors_with_capacity();

Check assignments:
  SELECT 
    c.name AS child_name,
    p.full_name AS doctor_name,
    cda.assigned_at
  FROM public.child_doctor_assignments cda
  JOIN public.children c ON c.id = cda.child_id
  JOIN public.profiles p ON p.id = cda.doctor_id
  ORDER BY cda.assigned_at DESC;

Check follow-up dates:
  SELECT 
    name,
    screening_status,
    observation_end_date,
    CASE 
      WHEN observation_end_date > NOW() THEN 'Upload locked'
      ELSE 'Upload available'
    END AS status
  FROM public.children
  WHERE screening_status = 'under-observation';
